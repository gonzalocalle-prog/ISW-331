# Rúbrica de Evaluación — Semanas 1–8

**Total: 100 puntos | Formato: Demo individual 15-20 min**

---

## 1. Explicación del Proyecto *(40 pts — más importante)*

| Puntaje | Criterio |
|---------|----------|
| **36–40** | Explica claramente el problema que resuelve, quién es el usuario, y las feature/s principales. Puede responder preguntas sin dudar. Conecta cada decisión técnica con una necesidad del proyecto. |
| **26–35** | Explica el problema y el usuario, pero las features no están bien definidas o no puede justificar alguna decisión. |
| **16–25** | Describe el proyecto vagamente. No puede conectar las decisiones técnicas con el propósito del producto. |
| **0–15** | No puede explicar qué hace su proyecto o para quién es. |

---

## 2. Demo en Vivo del Frontend *(30 pts)*

| Puntaje | Criterio |
|---------|----------|
| **27–30** | La aplicación corre sin errores. Muestra CRUD o flujo principal, autenticación funcional y rutas protegidas. Maneja estados de carga y error visiblemente. |
| **20–26** | La aplicación corre con errores menores. Autenticación funciona pero falta manejo de errores o rutas protegidas. |
| **10–19** | La aplicación corre parcialmente. Falta autenticación o el flujo principal está incompleto. |
| **0–9** | La aplicación no corre o solo muestra pantallas estáticas. |

---

## 3. Backend y API *(15 pts)*

| Puntaje | Criterio |
|---------|----------|
| **13–15** | Al menos 2-3 endpoints funcionales, con validación básica. El estudiante puede explicar cada endpoint y su propósito en el proyecto. |
| **8–12** | Endpoints funcionan pero sin documentación o sin justificación clara de para qué sirven en el proyecto. |
| **1–7** | Endpoints, sin documentación o no conectados al frontend. |
| **0** | No tiene backend. |

---

## 4. Arquitectura y Organización *(15 pts)*

| Puntaje | Criterio |
|---------|----------|
| **13–15** | Tiene diagrama C4 actualizado. El repositorio tiene estructura clara, linters configurados y README que describe el proyecto real. |
| **8–12** | Diagrama existe pero desactualizado. README básico. Linters configurados. |
| **1–7** | Falta diagrama o README vacío. Estructura de carpetas confusa. |
| **0** | No tiene diagrama, no tiene README, repositorio desorganizado. |

---

> **Nota:** Si el estudiante no puede explicar por qué tomó una decisión técnica, ese criterio no puntúa completo, aunque el código funcione. El objetivo es verificar comprensión, no solo que el código corra.
